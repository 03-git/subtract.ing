Governance

The agent reads, reasons, prepares. You sign. The boundary is cryptographic.

The problem

AI agents modify your system. They edit files, run commands, change state. Without governance, you have no way to know:

  • What changed since you last looked
  • Whether a prior session introduced drift you didn't authorize
  • Whether the current state reflects your decisions or a confused agent's guesses

The failure mode is confabulation cascade: a session "corrects" something incorrectly, the wrong state persists, the next session treats it as truth, and you spend hours clawing back ground that was already held.

The solution

A manifest of file hashes, signed with your Ed25519 key. Two scripts. No daemon, no service, no database.

boot.sh exit.sh │ │ ├─ what changed? ├─ update manifest ├─ is it signed? └─ sign └─ surface drift

At session start, boot.sh verifies the manifest signature, computes current hashes, and surfaces any drift. At session end, exit.sh updates the manifest with current state and signs it.

The primitive

ssh-keygen -Y sign and ssh-keygen -Y verify, shipped with OpenSSH since 2019. Every Unix system has it. No blockchain, no certificate authority, no cloud service.

Unsigned drift = changes you didn't authorize

Signature = Ed25519 attestation that you reviewed and approved

Who signs

You sign alone. The signing identity is a single Ed25519 key you control. Agents verify but never sign for authority. Trust paths should be predictable: one identity, one head to audit.

The public half of your key lives in ~/.ssh/authorized_signers. This is the trust set verification checks against.

What gets signed

A single manifest file containing one line per governed file: the sha256 hash and the path.

sha256:a1b2c3...  config/settings.json
sha256:d4e5f6...  scripts/handler.sh
sha256:g7h8i9...  memory/lookup.tsv
signed-at: 2026-04-11T14:00:00Z

The manifest itself is signed once. Verification is one signature check plus N hash checks.

The bootstrap

The signing tools sign themselves. boot.sh and exit.sh are both included in the manifest. A session that rewrites either script to skip checks produces a script whose hash no longer matches. The next boot catches the mismatch.

The chain of trust:

  • boot.sh verifies the manifest signature
  • boot.sh verifies its own hash against the manifest
  • boot.sh verifies exit.sh's hash against the manifest
  • Only then does boot.sh verify the governed files

A confabulating session that wants to subvert this would have to forge your private key. The architecture cannot defend against a compromised root. Nothing can.

What this enables

  • Confabulation cannot persist. A session without your private key cannot produce a valid signature. Wrong corrections don't survive.
  • Day-forward gains stick. What you sign at session end persists as verified state into the next session.
  • History is auditable. Each signoff produces a dated manifest. Your whole trajectory becomes a sequence of signed snapshots.

What this does not solve

  • In-session confabulation. A session can still confabulate during runtime. The architecture only prevents persistence.
  • Compromised key. If your private key is exfiltrated, the attacker can sign anything. Physical security and key rotation are the mitigations.

The kernel has had this since 2019. subtract just calls it.
