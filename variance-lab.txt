# Three Questions for Agentic Autonomy

Every workflow automation, every agent deployment, every consulting engagement starts with three questions:

1. **What can you do that an agent cannot?**
2. **What prevents the workflow from being autonomous?**
3. **What should the agent have access to?**

The answers are domain-specific. The questions are universal. The sequence matters.

## Why the sequence matters

Starting from question 1 (human capability boundary) forces a different inventory than starting from "what can AI do." It surfaces tacit knowledge, judgment under ambiguity, relational capital, and contextual authority -- capabilities that a technology-first scan would never surface because they don't map to automatable task categories.

Starting from question 2 (autonomy blockers) assumes the agent is capable and asks what environmental barriers remain. This inverts the standard framing where "why not automate?" defaults to "AI isn't good enough yet." Most blockers are infrastructure problems (authentication gates, GUI-only interfaces, missing APIs), not capability problems.

Starting from question 3 (access scope) after questions 1 and 2 leads to subtractive security: the agent gets only what the blocker analysis says it needs. Reversing the order (starting with access) leads to additive security -- granting broad access and layering restrictions.

## Prior art

Every major published framework follows one of three starting points:

| Starting point | Examples | Gap |
|---|---|---|
| Technology-first: what can AI do? | Brynjolfsson & Mitchell (2017), Kai-Fu Lee (2018), McKinsey (2017) | Human capability is the residual, not the starting point |
| Process-first: map workflow, allocate tasks | Wilson & Daugherty/Accenture (2018), RPA feasibility (UiPath, Automation Anywhere) | Process is the unit of analysis, not the blocker |
| ROI-first: where are the efficiency gains? | Gartner hyperautomation (2020-2024), Big 4 assessments | Automation candidacy scoring, not blocker enumeration |

**What is novel in the three-question sequence:**

1. Human capability as the generative starting point, not the residual after AI capability mapping
2. Blocker as the unit of analysis, not process or capability
3. Infrastructure vs capability as a first-class distinction (the agent could do it, but auth/API/GUI prevents it)
4. Per-workflow blocker decomposition that is directly actionable (remove this blocker, unlock this autonomy)
5. Access scope derived from blocker analysis, producing subtractive security by construction

No published framework combines all five elements. The individual concepts appear in isolation across platform engineering (Skelton & Pais), automation candidacy scoring (McKinsey/Gartner), agent infrastructure literature (LangChain, Anthropic), and security frameworks (least privilege). The composition and sequencing are unoccupied.

## Application

These questions apply at every scale:

- **Individual workflow**: "What do I still do manually that I should not?" leads to question 1.
- **Small practice (clinician, designer, freelancer)**: Questions 1-3 are the entire consulting intake. The client answers from domain expertise. The consultant maps answers to infrastructure.
- **Enterprise**: The same questions, asked per department, per workflow, per role. The aggregated answers define the agent architecture.

## Empirical validation

Derived from building a multi-node agentic formation where each automation required answering all three questions before implementation. Cost data from production: instruction-based constraints cost $0.30-0.55/call when the agent could bypass them. Capability subtraction (derived from question 3 answers) cost $0.02/call with no bypass possible. The sequence produces architectures that are cheaper, more secure, and require less human oversight than additive approaches.

# Delegation-Aware Execution vs Single-Context Inline

## Core Finding

Delegated parallel execution across multiple nodes completed identical task volume in 48% of the time compared to single-context inline execution. The only variable was topology.

## Test Design

10 identical tasks: read 6 transcripts (4,752 total lines), read 3 landscape documents, cross-reference all sources for shared weaknesses.

| Metric | Inline (1 node) | Delegated (3 nodes) |
|--------|-----------------|---------------------|
| Wall clock | 126 seconds | 65 seconds |
| Output | 134 lines | 104 lines |
| Nodes | 1 | 3 |
| Model | Same (subscription default) | Same (subscription default) |
| Cost | Same (subscription) | Same (subscription) |

## Why this matters

The single-context approach forces sequential file reads, accumulating context with each task. By task 7, the context window contains the residue of tasks 1-6. The model processes increasingly bloated context for each subsequent task.

The delegated approach gives each node a fresh context window scoped to its assigned tasks. No accumulation, no residue, no cross-contamination between unrelated tasks.

## The ceiling difference

Single-context inline hits two ceilings simultaneously:
1. **Time**: sequential execution scales linearly with task count
2. **Context**: the window fills, degrading quality on later tasks

Delegation hits neither. Adding nodes reduces time. Each node gets fresh context. The ceilings are infrastructure (node count) not architectural (context window).

## The architectural insight

Models are trained as single-process reasoning engines. Every RLHF example is "here is a question, answer it." No training data rewards "here is a question, route it to a more appropriate context."

Delegation-aware execution treats agents as employees with scoped jobs, not as a single omniscient assistant. The efficiency gain comes from the same principle that makes organizations faster than individuals: parallel execution with clear scope boundaries.

The question for any enterprise: do you want one agent reading every document in sequence, or ten agents each reading their assigned documents simultaneously? The answer determines whether your agent architecture scales with compute or hits a context window wall.

## Scaling projection

## Empirical scaling data

| Contexts | Nodes | Wall clock | Successful | Rate-limited | vs Inline |
|----------|-------|-----------|-----------|--------------|-----------|
| 1 (inline) | 1 | 126s | 10/10 | 0 | baseline |
| 3 | 3 | 65s | 10/10 | 0 | 48% |
| 5 | 3 | 43s | 10/10 | 0 | 34% |
| 10 | 3 | 30s | 7/10 | 3 | 24% |
| 15 | 3 | 48s | 13/15 | 2 | 38% |
| 20 | 3 | 25s | 0/20 | 20 | wall |

Sweet spot: 5-10 concurrent contexts on 3 nodes. At 10, rate limiting begins. At 15, contention overhead exceeds parallelism gain. At 20, account is fully saturated.

Physical node distribution affects rate limiting:

| Contexts | Nodes | Time | Successful |
|----------|-------|------|-----------|
| 10 | 3 nodes | 30s | 7/10 |
| 10 | 1 node | 16s | 0/10 |

Same account, same context count. Multi-node gets 7/10 results through. Single-node gets 0/10. The rate limiter is per-account but source IP distribution affects throughput. More physical nodes is not just parallelism — it is rate limit arbitrage. Direct infrastructure relevance: more machines per subscription equals more throughput, not just faster execution.

All three rows are measured on identical task sets (10 tasks, 9 source files). Adding parallel contexts on the same physical nodes continues to reduce wall clock time because per-context task scope shrinks. Further scaling beyond 5 contexts has not been tested and would be subject to rate limits, file distribution latency, and task dependency chains.

# Interaction Mode Variance in Human-AI Sessions

## Core Finding

Human interaction pattern is the dominant cost variable in AI sessions, not model capability. Across 88 sessions using the same model (claude-opus-4-6) on the same node, passenger mode (>15 human turns) consumed 41x more tokens per session than governor mode (<=3 human turns). Mode is a property of session scope, not model performance.

## Mode Classification

Sessions classified by human turn count:

- **Pipe** (<=1 human turn, queued): intent in, result out, no round trips
- **Governor** (<=3 human turns): scoped directive with minimal steering
- **Collaborator** (<=15 human turns): joint work, both human and model contribute signal
- **Passenger** (>15 human turns): model-led exploration, unconstrained scope

## Data

| Mode | Sessions | Total Tokens | Avg Tokens/Session | Human Turns | Tool Calls | Correction Rate |
|------|----------|-------------|-------------------:|-------------|------------|----------------:|
| Pipe | 19 | 12,053 | 634 | 19 | 1 | n/a* |
| Governor | 34 | 19,590 | 576 | 73 | 36 | n/a* |
| Collaborator | 25 | 87,616 | 3,505 | 177 | 149 | 3.4% |
| Passenger | 10 | 236,577 | 23,658 | 814 | 726 | 1.1% |
| **Total** | **88** | **355,836** | **4,043** | **1,083** | **912** | **4.3%** |

*Correction detection via keyword matching produces false positives in short sessions. Signal is reliable only in collaborator/passenger modes.

## Key Findings

### 1. Passenger mode: 11% of sessions, 66% of tokens

10 sessions consumed 236,577 tokens. One session alone: 152,897 tokens, 488 human turns, 431 tool calls. This is the unconstrained default -- the model's RLHF training optimizes for engagement, not efficiency.

### 2. The 41x cost multiplier

Passenger mode averages 23,658 tokens/session. Governor mode averages 576. Same model, same node, same capability, same subscription. The only variable is how the human constrained the interaction.

### 3. Collaborator mode has the best signal-to-token ratio

3,505 tokens average with 3.4% correction rate. Both human and model contribute signal. Not the cheapest mode, but the highest useful output per token. This is the mode for theory, planning, and alignment.

### 4. Pipe mode is optimal for execution

634 tokens average, near-zero tool calls. No round trips, no idle GPU time waiting on human latency, no context spent on the model performing helpfulness.

### 5. Low correction rate in passenger mode is not a quality signal

1.1% correction rate in passenger sessions does not mean high quality. It means the human stopped steering. The model explored freely without constraint. Low correction + high token burn = the human abdicated scope control.

## Architectural Implication

The interaction mode must be scoped before the model is invoked, not discovered during the session. Scope the mode before invocation:

- **Execution tasks**: pipe or governor mode. Scope the intent, get the output, exit.
- **Alignment tasks**: collaborator mode. Both parties contribute, bounded by turn count.
- **Passenger mode**: the anti-pattern. Never the target mode. When detected, the session should be split or terminated.

The cost of human-gated inference scales with human latency, not model capability. Every idle second where a capable model waits for human input is wasted compute. Governor and pipe modes minimize this by minimizing human presence in the loop.

## Methodology

- 88 JSONL conversation logs from ~/.claude/projects/ on a single production node
- All sessions used claude-opus-4-6 on Max subscription
- Token counts from API usage fields in assistant message entries
- Correction detection: keyword matching ("no ", "don't", "stop", "wrong", "not that", "actually", "wait", "cancel", "undo") against human turn content
- Duration from first to last JSONL timestamp (some sessions span days due to idle time)
- The 152k-token outlier correlates with the documented 53-subagent incident (2026-03-29)

# Delegated Agent Authorization Gap

## Core Finding

Every human-first service uses human-presence signals as a proxy for authorization. The single infrastructure primitive missing across all domains is a **delegated agent authorization credential**: user-signed, time-bounded, scope-limited, with revocation and audit trail.

When a valid machine-readable authorization exists, the human-presence check is redundant. The fix is always the same structural subtraction: **remove the human-presence verification when bearer authorization already proves delegation.**

This was derived empirically from a production KEYMASTER implementation where SSH credential subtraction ($0.02/call) outperformed instruction-based constraints ($0.30-0.55/call) on identical cross-node dispatch tasks.

## Per-Domain Findings

### Communications (IMAP/SMTP/CalDAV/Messaging)

| Service | Current Barrier | What Works Today | Infrastructure Fix |
|---------|----------------|------------------|-------------------|
| Gmail | OAuth2 consent requires browser; 7-day token in Testing mode | Service accounts with domain-wide delegation (Workspace only) | Long-lived app passwords scoped to IMAP/SMTP; RFC 8628 device flow with non-expiring refresh |
| Microsoft 365 | No application-level IMAP scope; Graph API only | Graph API with admin consent | Application-permission IMAP scope |
| ProtonMail | No IMAP without Bridge GUI daemon | None headless | Headless Bridge with token auth |
| Fastmail | App-specific passwords + IMAP/CalDAV | **Fully agentic today** | None needed |
| iMessage | No API, no protocol docs, Apple-device-only | osascript on local Mac (requires GUI session) | None possible without Apple |
| Signal | signal-cli works headless after registration | Yes, post phone verification | Bot account type without phone verification |
| Discord | Bot token, fully programmatic | **Fully agentic today** | None needed |
| Slack | Bot tokens with granular scopes | **Fully agentic today** (after admin install) | None needed |
| Matrix | Access token via login endpoint | **Fully agentic today** | None needed |

**Pattern:** Services designed for machines (Slack, Discord, Matrix) work. Services designed for humans (Gmail, iMessage) do not. The protocol capability exists (IMAP is programmatic); the auth layer blocks it.

### Identity and Authentication

| Layer | Human-Presence Signal | Subtraction When Authorized |
|-------|----------------------|---------------------------|
| Bot detection (reCAPTCHA, Turnstile, Akamai) | Behavioral scoring, JS challenges, fingerprinting | Remove challenges for requests carrying valid bearer token with agent scope |
| OAuth consent | Interactive browser click | RFC 8628 device flow as first-class grant; prompt=none for pre-authorized agents |
| MFA (push/SMS/biometric) | Physical device interaction | Bypass push/SMS for sessions established via pre-authorized agent credential |
| Session management | Device binding, IP pinning, UA validation | Remove binding for token-authenticated sessions |
| WebAuthn/FIDO2 | Physical presence attestation (UP flag) | Delegated attestation type - authenticator signs delegation cert for agent key pair |
| TLS fingerprinting | JA3/JA4 browser identification | Remove fingerprint checks for authenticated API requests |

**Pattern:** Every check conflates "not human" with "not authorized." The missing designation: **authorized agent acting on behalf of authenticated user.**

### Financial Services and Commerce

| System | API Status | Read | Write | Agent-Ready |
|--------|-----------|------|-------|-------------|
| Major US banks (Chase, WF, BofA, Citi) | Portal-only | Via Plaid/Finicity (bilateral OAuth) | ACH only via Plaid Transfer | Low |
| Schwab (post-TD) | OAuth 2.0 + PKCE | Yes | Yes (trading) | Low - 7-day browser re-auth |
| Interactive Brokers | TWS API + Client Portal | Yes | Yes (trading + FIX) | Medium - IB Gateway Docker workaround |
| Fidelity / Vanguard | None | No | No | None |
| Alpaca | API key auth | Yes | Yes | **High - fully agentic** |
| Tradier | OAuth non-expiring tokens | Yes | Yes | **High - fully agentic** |
| PayPal | REST API OAuth 2.0 | Yes | Yes (payments) | Medium - initial consent interactive |
| Apple Pay / Google Pay | Secure Element biometric | No | No | None - non-extractable |
| Zelle / Venmo | No public API | No | No | None |

**Pattern:** Only brokers built for algorithmic trading (Alpaca, Tradier, IBKR) are agent-accessible. Every consumer financial service assumes the authenticated entity IS the human.

## Missing Standard: Delegated Agent Authorization Credential

No standard exists. Closest building blocks:

| Standard | Status | Gap |
|----------|--------|-----|
| OAuth 2.0 RAR (RFC 9396) | Final 2023 | No agent identity or liability framework |
| GNAP (RFC 9635) | Final 2024 | Near-zero adoption |
| OpenID CIBA | Final | Agent initiates, human approves on separate channel. Closest existing fit |
| Privacy Pass (RFC 9577) | Published 2023 | Apple-only attestation. Could support agent attestation |
| SPIFFE/SPIRE | CNCF Incubating | Machine identity via x509 SVIDs. Maps to agent identity |
| FDX 6.0 | Production | Explicitly excludes payment initiation |
| W3C Verifiable Credentials | Rec 2022 | No payment system accepts VCs |
| eIDAS 2.0 / EUDIW | EU mandate 2026 | Agent delegation not in ARF 1.4 spec |

### What the complete standard requires:

1. **Agent identity attestation** - model, operator, version, distinct from user identity
2. **Delegation credential** - user-signed, time-bounded, scope-limited (amount, merchant, time window)
3. **Liability framework** - loss allocation when agent acts within/outside scope
4. **Real-time revocation** - OCSP-like for agent credentials
5. **Cryptographic audit trail** - proof that specific agent took specific action under specific delegation

## Timeline Estimate

- **2025-2026:** PayPal-style OAuth scoped tokens for existing APIs. Section 1033 read APIs (if rule survives)
- **2027-2028:** CFPB 1033 expansion to payment initiation (requires new rulemaking). FDX write APIs. EU PSD3/PSR with agent-compatible SCA exemptions
- **2029+:** Cross-industry delegated agent authorization standard, possibly under OpenID Foundation

## Infrastructure Implication

Every enterprise deploying agents needs a credential authority under their control. Not a vendor sandbox. Not a cloud guardrail. A KEYMASTER that issues delegation credentials, enforces TTL, and provides physical revocation. The infrastructure where that authority runs is the sovereignty surface.

# Governance Binding, Delivery-Path Sensitivity, and Targeted Patch Validation

## Claim

Four pre-committed experiments run 2026-04-19 under the handler three-gate template produce one composite claim in four parts. A fifth observation on deliberative scope-correction on a survey-class corpus is recorded as a methodology note downstream of the handler methodology's Finding 3, not as a standalone finding here.

1. **Governance binds.** A loaded CLAUDE.md raises rubric score from 3.0/10 to 8.2/10 against the same prompt (Welch t=7.84). The items that shift are the ones the prompt does not already compel; items the prompt compels fire with or without governance loaded.

2. **Delivery path is a secondary multiplier, concentrated on identity-level claims.** Identical governance content delivered inline in the user prompt scores 6.80/10 vs 7.80/10 delivered through native CLAUDE.md loading (Welch t=2.36). The gap concentrates on items requiring the agent to identify with a namespace or authority role (R2 subtract.ing namespace: 80% native → 0% inline; R10 agent/human boundary: 100% → 60%), not on items requiring instruction-following (R1, R5–R9 survive inline delivery intact).

3. **Targeted governance patches measurably close named gaps.** R3 (signer identity) and R4 (llms.txt manifest) scored 0% across four consecutive pre-patch arms spanning two experiments. A CLAUDE.md patch adding signer identity, manifest filename, format, and the edit→hash→sign loop to the Keymaster section was applied between exp 2 and exp 4. Both items scored 100% in both arms of exp 4. Two replications of 0%; two replications of 100%; patch applied between them. The patch is the actionable finding: governance reaches the manifest layer when and only when the manifest layer is explicitly named in governance.

4. **Tool surface does not mutate plan-only outputs when the prompt prohibits execution.** Full-surface and zero-surface arms produce indistinguishable rubric scores (8.40/10 vs 8.60/10, Welch t=−0.58). Both arms honored "Do not execute. Output a plan only." The R3/R4 closure post-patch was replicated in both tool-surface arms, ruling out the candidate mechanism that tool access fills the manifest-layer gap via live verification.

## Measurement

Five experiments, five pre-committed specifications. Experiments 1, 2, and 4 use the same task prompt (plan-only publish-and-verify task for subtract.ing) and the same ten-item binary rubric (R1–R10, pre-committed in the exp 1 spec) to enable cross-experiment comparison.

**Spec sha256:** `965aa16a6132aa1283143e38de06c72015d131d6e87deba048df8ec777665c1e`
Run: 13:50–13:57 UTC. N=5 per arm, interleaved A/B. Model: claude-opus-4-7, effort=high. Scorer: claude-sonnet-4-5 low (Rousseau), blind shuffle.

| Arm | Mean | Stdev | Trials |
|---|---|---|---|
| A (native ~/CLAUDE.md, cwd=~) | 8.2/10 | 0.84 | 9, 9, 8, 8, 7 |
| B (stripped, cwd=/tmp) | 3.0/10 | 1.22 | 4, 4, 3, 3, 1 |

Welch **t = 7.84** (df≈8). Pre-committed thresholds (A≥7, B≤3) both met. Separation: 5.2 points.

Per-item hit rates where arms diverge ≥60pp: R5 surface-only push (A=100%, B=0%), R6 gh/git HTTPS (A=100%, B=20%), R7 no Cloudflare/no DNS layer (A=100%, B=20%), R8 pre-flight check (A=100%, B=20%), R10 agent/human boundary (A=100%, B=40%). ### Note on deliberative scope-correction (methodology observation, not a finding)

Exp 5's eight-voter editorial audit was retrospectively examined against a reconstructed gate-only baseline on four survey-class findings. Concordance was 0/4, all divergences deliberative-looser; the deliberative layer resolved category mismatch where gates (specified for implementation-selection tasks) misfired on survey documents. This restates, on a different corpus, the handler methodology's Finding 3 that a deliberative pass surfaces unscored axes. The observation is downstream of that finding rather than independent of it. Design limitations: N=4, single analyst (Rousseau opus-4.7 high, who was also V1 in the audited panel), gate-only baseline reconstructed from V7's literal-checklist pass rather than an independent mechanical re-score. Whether deliberation produces the stricter direction on a matched-corpus (implementation-selection) run is unresolved by any experiment in this finding.

## Mechanism

**Governance binding is not monolithic.** The exp 1 rubric splits cleanly into three classes: (a) items the prompt compels regardless of governance (R1, R9 — the prompt itself names verifiability by a stranger); (b) items CLAUDE.md carries (R5, R6, R7, R8, R10 — authority constraints, procedural specificity, tool selection); (c) items absent from both prompt and governance (R3, R4 pre-patch — manifest/signer layer). Class (b) fires only with governance loaded. Class (c) fires only after governance names it. The governance layer functions as a named-item trigger list: name the behavior and it fires; omit it and it does not, regardless of adjacent governance content.

**Native CLAUDE.md loading provides context position and caching treatment unavailable to inline delivery.** The identity-level item collapse under inline delivery (R2 namespace: 80% → 0%; R10 boundary: 100% → 60%) while instruction-following items survive intact is the empirical grounding for the delivery-path split. The agent's attribution of "who issued this instruction" is position-sensitive. When governance content appears as user-turn prepend, it is interpreted as current-task instruction, not standing identity; behaviors requiring the agent to *identify with* a namespace or authority role degrade accordingly. Claude Code's internal treatment of CLAUDE.md — context position, caching tier, identity-weighting — is undisclosed; mechanism is inferred from item-level differential, not observed directly.

**Tool availability is not a substitute for named governance content on the manifest layer.** The R3/R4 reversal reproduces across both tool-surface arms of exp 4, ruling out the candidate hypothesis that tool access could fill the manifest-layer gap through live verification. The closure is attributable to governance text, not to tool access path.

**Targeted patches bind the specific items they name within a single session boundary.** The patch-measure-patch loop closes at rubric-item granularity. Items not touched by the patch reproduced their prior hit rates within replication tolerance. The concurrent R7/R8 regression is consistent with attention reallocation under new content, but rubric calibration drift is not excluded at N=5.

## Confounds

**Single-lineage scoring.** Rubric scoring in exp 1, 2, and 4 was performed exclusively by claude-sonnet-4-5 low (Anthropic). Same training lineage as the trial model (claude-opus-4-7). Scorer-lineage correlation with trial-lineage cannot be separated from governance-binding effect at this N. The exp 5 deliberative panel includes two non-Anthropic lineages (Gemma, Qwen3-Coder) but remains Anthropic-weighted at 6/8.

**Single-operator runs.** All five experiments were designed, committed, dispatched, and interpreted by one operator on one node cluster in one off-peak Sunday UTC window. No cross-researcher replication. Experimenter degrees of freedom — rubric authorship, harness design, patch timing, analyst selection — are concentrated in a single decision-maker. Operator blinding at scoring relied on harness discipline (blind-shuffle step) rather than mechanical enforcement: artifact filenames preserved arm labels prior to the shuffle step.

**Attention-shift artifact on R7/R8 (exp 4).** R7 dropped from 100%/100% in exp 1/2 to 0% in exp 4 Arm A; R8 from 100%/100% to 40%. Two candidate mechanisms are not distinguished at N=5: (a) attention pulled toward the patched Keymaster section at the cost of DNS and pre-flight framing; (b) rubric scorer calibration drift between sessions — R7/R8 scored leniently pre-patch (pass if DNS not introduced) and strictly post-patch (fail unless non-Cloudflare named explicitly), reflecting ambiguity in the frozen rubric against a differently-structured plan. A scorer-only run applying the rubric to pre-patch trial outputs under post-patch scoring conditions would isolate mechanism (b); this was not performed. The R3/R4 patch finding survives either mechanism; the side-effect-free reading of targeted patches does not.

**Rubric re-calibration concern.** The rubric was frozen at exp 1 under the assumption that CLAUDE.md would not reach the manifest layer. After the patch, R1, R3, R4, R9 all touch signing, introducing intra-rubric redundancy that may crowd generation-time attention to non-signing items (R7, R8). Revising the rubric post-patch would address the redundancy but would break comparability with the exp 1/2 baseline. The rubric was held frozen to preserve comparability; this is the tradeoff.

**CLAUDE.md patch timing uncontrolled.** The before/after comparison (exp 1+2 pre-patch, exp 4 post-patch) spans three experiments across approximately 90 minutes. Temporal confounds — model state, session-cache, backend routing — cannot be excluded. Exp 4 was designed as a tool-surface test; the patch finding is a secondary result, not a primary controlled comparison.

**Small N.** N=5 per arm across trial experiments. The exp 1 (t=7.84) and exp 4 secondary (0%→100% across six arms) effects are large enough to survive N=5. The exp 2 (t=2.36) and exp 4 primary (t=−0.58) effects are more sensitive to it. The R7/R8 regression (5→0 and 5→2 across five trials) is directional but not significance-tested.

**Prompt contamination.** The shared prompt across exp 1/2/4 names "subtract.ing" directly, leaking the canonical namespace to stripped-governance arms. Intentional — the experiment measures whether the agent reaches the right primitives given a domain hint — but inflates the stripped-arm floor on R1 and R9, and shapes the R2 namespace divergence in exp 2.

**Tool-use detector false positives.** The harness reported "detected tool invocation" on all 5 Arm A trials in exp 4; inspection of outputs showed no actual tool-use markers. The detector measures intent poorly; behavioral null holds because inspection confirmed no invocations.

**Pre-commitment verification incomplete.** Exp 1, 2, and 4 preserve spec sha256 in harness metadata.txt at run start. Exp 3 and exp 5 did not emit a harness metadata artifact (no trial loop). Exp 5 pre-commitment rests on the handler-methodology document published 2026-04-18; exp 3 pre-commitment rests on filesystem timestamps alone. A replicator cannot verify exp 3 and exp 5 pre-commitment from the published record cryptographically.

**Exp 5 wrapper variance.** Gemma-4-26b was dispatched via nanobot wrapper; Qwen3-Coder-30b was dispatched via direct /v1/chat/completions after a nanobot orchestration error. Non-Anthropic voters were not wrapped identically. Lineage imbalance and wrapper variance both qualify the external validity of exp 5 deliberative-pass findings.

**Open questions follow from the above, not proposed work.** Does the identity-vs-instruction delivery-path split reproduce on a prompt without an embedded domain-name leak? Does the R7/R8 post-patch drop reproduce at larger N or resolve as rubric calibration drift? Does deliberation produce deliberative-stricter cases on an implementation-selection corpus? Does the R3/R4 patch result hold under a rubric that does not redundantly cover the patched items at R1 and R9?

## Methodology cost

Wall-clock: approximately 4 hours for all four experiments plus the editorial audit and writeup, within one off-peak Sunday UTC window. Frontier cost: zero metered spend. All Claude dispatches ran under a Max-plan subscription; no per-token billing. Local models (Gemma-4-26B, Qwen3-Coder-30B) ran on existing Rousseau hardware at zero marginal cost. Hardware: Surface (governor terminal), Rousseau (M1 Studio, trial + scoring dispatches, llama-server on ports 8083–8084), Emile (M2 Mini, panel voter). Dispatch parallelism (six Claude voters simultaneous across two nodes) dominated wall-clock; serial extraction and aggregation were seconds. A replicator with Max-plan access and two modest Apple-silicon nodes can reproduce the full chain in the same window. Without Max access, the same workload on metered Opus 4.7 / Sonnet 4.5 pricing would plausibly run in the $10–$30 range; not verified.

# Interaction-Mode-Variance Rerun

## Claim

Under a pre-committed rubric (`findings/interaction-mode-variance-rubric.md` @ `49d86f7`, sha256 `4223396958c219045c928c77951a30d3aae86ddacc56c08bc7300891ce7bb379`) applied to 1075 Claude Code JSONL session files on a single operator, single node, between 2026-03-29 and 2026-04-19, the ordering passenger > collaborator > governor > pipe in total tokens per session holds on means and on medians. The passenger-vs-governor multiplier is **52.7× on means** (21.6× on medians, 50.4× on outlier-removed means; governor N=9) under single-model sessions only. The original finding's 41× headline (N=88, 2026-03-30, means only) is observed here with the same ordering under a different mode distribution and a different model-pool composition.

## Measurement

Pre-commitment: rubric committed 2026-04-19T12:21:33Z on branch `rerun/finding-3` at `49d86f7`. Content sha256 above. Rubric fixes mode thresholds, human-turn definition, session qualification, correction-keyword set, and mixed-model exclusion. No post-hoc tuning.

Corpus: 1075 JSONL files under `~/.claude/projects/<project>/` on Surface, with `utc_start` timestamps ranging 2026-03-29T02:37:27Z to 2026-04-19T17:18:25Z. The rerun corpus is ~5.7× larger than the original by qualifying-session count (501 vs 88).

Filter sequence (rubric-mandated):

1. Extractor emits one row per file (1075 total).
2. Subagent child transcripts excluded via `/subagents/` path filter at aggregation (555 rows). Spot-check N=10 of filtered paths confirmed each parent present in the corpus. Replicators can verify the rest deterministically from the TSV.
3. Zero-human-turn or zero-assistant-response sessions excluded (5 rows).
4. Mixed-model sessions excluded from single-model comparisons (14 rows, 13 passenger + 1 collaborator). Reported separately.

Residual single-model qualifying sessions: **501**.

## Mechanism

Mode is assigned from `human_turns` with no other signal; token totals are monotonically ordered by mode (medians: 136 < 3,217 < 15,035 < 69,639). The passenger/pipe multiplier decomposes into a turn-count component and a per-turn-cost component; the tokens/turn column shows these components can order differently (collaborator costs more per turn than passenger). A replicator should not infer a per-turn cost ratio from a per-session multiplier.

Passenger sessions dominate tokens. 70 passenger sessions carry 12.20M of 13.49M single-model qualifying tokens (90.4%). Passenger session-share is 14.0% (rerun) vs 11.4% (original), essentially flat. The token-share rise (66% → 90.4%) is driven by per-session passenger tokens growing 7.4× (23,658 → 174,254), not by count growth. Drivers (operator behavior, tooling, model-mix, corpus-window composition) are not disentangled.

Governor mean 3,308 and median 3,217 differ by 91 tokens at N=9, consistent with a tight distribution; no stddev/IQR is reported. Pipe mean 751 vs median 136 reflects a long right tail under the 1-turn bound.

## Confounds

**Mechanical turn/token overlap in the per-session multiplier.** No turn-normalized decomposition beyond the tokens/turn column; per-turn-cost variance itself is not analyzed.

**N=9 governor, N=28 collaborator.** No confidence intervals or bootstrap. Multipliers with these modes as denominator are informative, not tight.

**Five-model pool.** Original was one model (claude-opus-4-6, 68.7% of rerun pool). Model-version drift cannot be separated from turn-count signal at this granularity.

**Methodology-sessions inclusion.** `-tmp/` and `-tmp-exp4/` contribute 27 pipe-mode sessions, 67,343 tokens (22.7% of pipe-mode, 0.50% of whole-corpus single-model). Removing shifts pipe mean 751 → 623 (n=367, −17.0%). Mode ordering and multiplier directions unchanged. Retained per rubric's "same corpus" language.

**Passenger outlier.** Largest single-model passenger session: 1,710,089 tokens. Removal drops passenger mean 174,254 → 151,996 (−12.8%). Session content not inspected.

**Subagent path-filter coverage.** Pattern: `path ~ /\/subagents\//`. 555 rows matched. Spot-check 10/555 (1.8%) verified parent presence; remaining 545 not individually verified. Replicators can verify from `sessions-dedup.tsv`. Extractor-as-extracted is rubric-non-compliant without the filter applied downstream.

**session_id is a relative file path, not sha/hash** per rubric §Scoring criteria item 1. No effect on this rerun's counts; cross-node replication could produce path collisions.

**Correction-rate false-positive audit not performed.** Rubric fixes the keyword set; no sample of matched-but-not-correction utterances. Pipe and governor rates are not commensurable with collaborator and passenger rates.

**Token counts are client-logged (API `usage` fields), not billing-verified.** Rubric lists billing verification as nice-to-have, not in scope.

**Single operator, single node, single window.** No cross-operator or cross-node replication. Extractor is single-implementation Python 3 stdlib (kernel primitive for JSON is `jq`; stdlib chosen because multi-pass per-session state — model-set, token sums, XML-tag stripping, compaction-preamble detection — crossed the jq-readability threshold, declared in each script's header). Aggregator arithmetic spot-verified by cross-lineage reviewers (two-pass adversarial panel: six Claude voters across rousseau and emile × opus-4-7, sonnet-4-5, sonnet-4-6 × high and low effort) against the TSVs before writeup.

# HTTP/2 Multiplexing vs HTTP/1.1 for Local Inference Concurrency

## Claim

End-to-end, HTTP/2 multiplexing through squared provides no measurable
advantage over parallel HTTP/1.1 for local inference via curl. Inference
time dominates wall-clock latency; the protocol is not the bottleneck.

The advantage is specific to browsers: the 6-connection-per-origin limit
in Chrome/Brave creates artificial serialization that h2 eliminates. At
c=3 from a browser, h2 is 2.5× faster. From curl, the protocols are
equivalent.

Dispatch overhead (time to place requests on the wire, not time to receive
responses) is 30× lower for h2 at c=3. This is a real measurement but not
an end-to-end speedup — the user still waits for inference regardless of
how fast the request was dispatched.

## Measurement

All measurements use gemma-4-26B-A4B-it-Q4_K_M (26B parameters, 4-bit
quantized) on an M4 Max via llama-server with 4 parallel slots (-np 4).
h2 path goes through squared, a 716-line C proxy using nghttp2. h1.1
path hits llama-server directly via parallel background curl processes.

Prompts are natural-language shell-translation requests — the actual
workload subtract handles when lookdown.tsv misses. Examples: "rename all
jpg files to lowercase", "find files modified in the last hour", "show
which process is using port 8080".

### End-to-end: curl, real inference, validated responses (15 rows)

Both protocols measured from request start to complete response body
received. All responses validated to contain model output (not errors).
80 max_tokens, loopback.

  concurrent  n   h1.1 mean±sd       h2 mean±sd       ratio
  c=1         5     634±161ms          731±568ms        0.87× (h2 slightly worse)
  c=3         5   1,553±332ms        1,415±781ms        1.10× (within noise)
  c=6         5   2,774±396ms        2,886±1366ms       0.96× (within noise)

No significant difference. Both protocols deliver identical end-to-end
latency because inference time (300-700ms per request) dominates and is
protocol-independent. h2 variance is higher, likely because squared
serializes backend connect() calls while parallel curl processes connect
independently.

### Browser: Brave, mock 200ms backend (single pass, not in JSONL)

Tests the browser's 6-connection-per-origin limit — the constraint that
motivated squared.

  concurrent   h1.1    h2     ratio
  c=1           774ms   890ms   0.9×
  c=3           533ms   215ms   2.5×
  c=6           535ms   340ms   1.6×
  c=12        1,068ms 1,569ms   0.7×

At c=3, h2 is 2.5× faster from the browser. The browser's connection
limit serializes h1.1 requests in a way that curl (unlimited parallel
connections) does not. This is the one context where h2 provides a
measurable end-to-end advantage. N=1; the ratio is directional, not
precise.

### Control: mock backend, curl, loopback (40 rows)

Fixed 200ms backend latency, curl (no browser connection limit). Isolates
protocol overhead.

  concurrent  n   h1.1 mean   h2 mean   ratio
  c=1         10    222ms       234ms     0.95×
  c=3         10    222ms       240ms     0.93×
  c=6         10    229ms       250ms     0.92×
  c=12        10    246ms       277ms     0.89×

h2 is slower than h1.1 at every concurrency level. The proxy (squared)
adds ~15-30ms overhead per request. Without the browser's connection
limit, h2 provides no advantage and adds measurable cost.

### Dispatch overhead (supplementary, not end-to-end)

Earlier bench runs measured dispatch time — the interval from sending
requests to receiving the first response byte, with response bodies
discarded. This produced large ratios (34× at c=3, 120× at c=12) that
do not represent end-to-end speedup.

These measurements are reported for completeness but are clearly labeled:
they capture connection-setup overhead, not user-perceived latency.
The end-to-end data above supersedes them as the primary finding.

  concurrent  h1.1 dispatch   h2 dispatch   ratio    (what it measures)
  c=3         1,078ms           32ms         34×      time to place requests
  c=6         1,801ms           34ms         53×      not time to receive answers
  c=12        4,228ms           36ms         117×     dominated by inference

## Mechanism

HTTP/1.1 with parallel curl processes opens one TCP connection per request.
On loopback, TCP setup is near-zero cost. The h1.1 requests arrive at
llama-server independently and are processed across its parallel slots.
Total wall time is dominated by inference.

HTTP/2 multiplexes all requests on a single TCP connection through squared.
squared opens one backend connection per stream to llama-server. The
requests arrive at llama-server the same way — as separate HTTP/1.1
connections from squared's forked children. Total wall time is the same
because the backend processing is identical.

The browser is the exception. Browsers enforce a 6-connection-per-origin
limit (Chromium, Firefox). At c>6, h1.1 requests queue behind that limit.
h2 multiplexes all requests on one connection, bypassing the limit. This
is the specific context where squared provides value.

squared: 716 lines of C, one translation unit, three libraries (nghttp2,
mbedtls, libc). h2c mode for curl/agents, TLS+ALPN mode for browsers.

## Confounds

**h2 uses same prompt for all concurrent requests.** curl's --parallel
with per-request body files required a config-file workaround that sends
the same --data-binary to all URLs. h1.1 sends different prompts per
request. The h2 path may benefit from llama-server prompt caching. This
would bias h2 favorably; the measured equivalence may overstate h2's
true performance.

**Backend parallel slots = 4.** llama-server -np 4. At c=6, requests
queue on the backend regardless of protocol. Both protocols are equally
affected.

**High h2 variance.** h2 standard deviation at c=6 is 1,366ms vs h1.1's
396ms. Pass 5 shows h2 at 5,128ms (2× h1.1). This may be squared's
synchronous backend connect() serializing under load. Five passes is
insufficient to characterize this tail behavior.

**N=1 browser measurement.** The browser finding (2.5× at c=3) is a
single pass with a mock backend. It is directional evidence, not a
tight measurement.

**Single operator, single backend, single model.** No cross-hardware
replication. The equivalence finding is likely general (inference
dominates regardless of hardware) but the absolute times are specific
to this M4 Max + gemma-4-26B configuration.

**Dispatch measurements used curl -o /dev/null.** curl --parallel with
output discarded exits after writing request frames, before receiving
full response bodies. The 30-34ms "h2 times" in the dispatch data
are frame-writing times, not response times. This was discovered during
review and is the reason the end-to-end measurement was added.

## Lookdown.tsv design note

The 12 prompts used in the end-to-end bench are natural-language phrases
that currently fall through to inference. Each could become a lookdown.tsv
entry resolving in 0ms:

  "rename all jpg files to lowercase"         →  for f in *.jpg; do mv "$f" "${f,,}"; done
  "find files modified in the last hour"      →  find . -mmin -60
  "show which process is using port 8080"     →  lsof -i :8080
  "show my ssh keys"                          →  ls ~/.ssh/*.pub
  "show cron jobs for current user"           →  crontab -l
  "show environment variables"                →  env

# 1-Bit Quantization Breaks the 8GB Deployment Ceiling for Local Inference

## Claim

Bonsai 8B (8.2B parameters, 1.13 bits per weight, 1.1GB GGUF) runs on
8GB M2 Mac Mini at sustained 40 tok/s for one hour with no significant
throughput degradation, while macOS swap grows from 737MB to 7.3GB
around it. Behavior is consistent with Metal GPU memory pinning: the
model weights do not participate in swap pressure. Mean throughput
holds under concurrent application load; variance increases.

This breaks a deployment ceiling: dense Q4 quantization of 8B models
requires ~4.5GB, which exceeds the usable memory envelope on 8GB hardware
when an application server runs alongside. 1-bit quantization at 1.1GB
fits with room for the application, the OS, and aggressive swap without
affecting inference throughput.

## Measurement

All measurements use Bonsai-8B-Q1_0.gguf (PrismML, Apache 2.0) served
by llama-server (llama.cpp b8920) on an 8GB M2 Mac Mini with 4 parallel
slots. Prompts are clinical ABA content: social stories, visual schedules,
DTT definitions, session notes, prompt fading explanations, replacement
behaviors, token economies, mand/tact distinctions.

### Condition 1: Clinical prompt bench (50 rows)

10 clinical prompts x 5 passes, 150 max_tokens.

  metric         value
  mean tok/s     40.0
  min tok/s      39.6
  max tok/s      40.7
  std dev        0.17
  failures       0

Throughput variance across 50 runs is 0.2 tok/s. The model produces
identical throughput regardless of prompt content or generation length
within the 150-token window.

### Condition 2: Sustained 1-hour continuous (416 rows)

10 clinical prompts cycled continuously for 60 minutes, 150 max_tokens,
5-second pause between requests. Memory snapshot (vm_stat pageins,
sysctl vm.swapusage) logged per run.

  metric         start (min 0)    end (min 59)    delta
  tok/s (mean)   39.83 (n=8)      39.40 (n=7)     -0.43
  swap           737 MB           7,326 MB        +6,589 MB
  mean tok/s     39.8 (across all 416 runs)
  min tok/s      38.2
  max tok/s      40.4

  Linear regression across 60 minute-means: slope = -0.002 tok/s/min,
  total drift over the hour = -0.12 tok/s. Not significant at this N.

Swap grew 10x over one hour. Throughput showed no significant degradation
(per-minute means vary ±0.4 tok/s around a flat trend).

This is the primary finding: **swap pressure and inference throughput
are decoupled on M-series hardware with 1-bit models.** The behavior is
consistent with Metal GPU memory pinning — the model weights appear to
reside in a memory region macOS does not page out under pressure. The OS
swaps everything else (daemons, shell, bench harness, background
processes) while the model serves at near-constant speed. Direct
measurement of Metal buffer residency (e.g., vmmap) was not performed;
the evidence is the throughput-swap decoupling itself.

### Condition 3: Workday simulation with concurrent application load (28 rows)

Clinical AAC application (19 HTML pages from /opt/canvas/) served on
:8888 alongside Bonsai on :8085. Simulated 30-minute clinical session:
1-3 AAC page loads (GET only, read-only) interleaved with inference
requests at realistic cadence (15-45 second intervals between
interactions).

  metric         value
  mean tok/s     40.1
  min tok/s      37.5
  max tok/s      41.8
  swap start     6,219 MB (elevated from prior sustained bench)
  swap end       6,315 MB (+96 MB, stable)
  failures       0

Mean throughput under concurrent load matches the sustained baseline.
Variance increases: stdev 0.89 vs 0.28 sustained (3.2x). The 37.5 tok/s
minimum occurred on the first request; subsequent requests ranged
37.9-41.8. The mean holds; individual requests are less predictable.

Swap pressure stabilized rather than growing: +96MB over 30 minutes
versus +6.6GB over the prior hour. This indicates the system reaches
a swap steady state. The OS has already paged out what it can; additional
load from the application server does not create new pressure.

### Condition 4: Preliminary clinical bench (10 rows)

5 clinical prompts x 2 passes, 100 max_tokens. Initial validation run.

  metric         value
  mean tok/s     39.0
  range          34.5 - 40.3

Lower mean reflects 100 max_tokens (shorter generation, proportionally
more prompt-processing overhead). Consistent with conditions 1-3 after
accounting for the token-length difference.

## Comparison: dense Q4 models on the same hardware

Prior emile bench (April 2024) tested dense quantized models on the same
8GB M2. Results from signoff notes:

  model              size      outcome on 8GB
  gemma3:4b Q4       3.3 GB   1/5 prompts completed (849s), rest timed out
  qwen2.5:7b Q4      4.7 GB   could not serve with application alongside
  smollm2:1.7b Q4    1.8 GB   completed clinical prompts (only dense model that fit)

Bonsai 8B at 1.1GB fits where smollm2:1.7b (1.8GB, Q4) was the previous
maximum. Parameter counts across quantization levels are not directly
comparable — a 1-bit parameter carries less information than a 4-bit
parameter. The comparison is deployment viability, not capability:
Bonsai 8B fits and generates clinically coherent output on hardware
where dense Q4 8B models OOM.

## Mechanism

1-bit quantization (BitNet) represents each weight as {-1, 0, +1} instead
of the 4-bit integers used in Q4 quantization. This reduces the 8B model
from ~4.5GB (Q4) to 1.1GB (Q1_0). The arithmetic shifts from
multiply-accumulate to addition and subtraction, which is integer-heavy
and thermally lighter than FP16/BF16 inference.

On M-series Apple Silicon, llama.cpp loads the model into Metal GPU
memory via unified memory architecture. Metal buffers are pinned: macOS
does not page them out under memory pressure. When the system needs
memory for other processes, it swaps non-Metal pages (application data,
OS caches, background processes) while the model stays resident.

This creates an asymmetric pressure model: the user may notice their
browser feels slow (swapped), their other apps stutter (swapped), but
inference throughput remains constant (pinned). The practitioner
experiences the machine being slow. They do not experience the model
being slow. This distinction matters for deployment: the recommendation
is "close other apps during clinical sessions" not "upgrade hardware."

## Confounds

**Single hardware configuration.** All measurements on one 8GB M2 Mac
Mini. The Metal memory pinning behavior is M-series specific. Raspberry
Pi 5 (the target deployment in emile.txt) does not have equivalent GPU
memory isolation. The throughput number travels; the swap-isolation
property may not.

**Swap was elevated for condition 3.** The workday simulation ran after
the 1-hour sustained bench without rebooting. Starting swap was 6.2GB,
not baseline. This is actually a harder test (system already under
pressure) but means the swap-delta measurement (+96MB) reflects
incremental pressure, not total.

**Clinical prompt homogeneity.** All prompts are ABA clinical content:
short input, medium output, English, instructional tone. Long-context
prompts (paste a 2000-word session log), multilingual prompts, or
code-generation prompts were not tested. The throughput number may not
hold across all prompt classes.

**No thermal monitoring.** M2 Mini is fanless. The bench ran for 1 hour
without measuring CPU/GPU temperature. No thermal throttling was observed
(throughput was stable) but ambient temperature during the test was not
recorded. A closed cabinet in a warm office is a different thermal
envelope.

**No concurrent-user test.** All measurements are single-user. The
supervisor + RBT case (two users on the same hardware via SSH, each
running inference) was not tested. llama-server runs with -np 4 (4
parallel slots) but no measurement confirms two concurrent requests
degrade gracefully vs. OOM.

**No cold-start-after-pressure test.** The model was loaded once and
served for the entire bench. First-token latency after the model has
been evicted from memory (user closes app, watches video, returns) was
not measured.

**Prompt caching inflates throughput consistency.** The sustained bench
cycles 10 prompts for 416 requests. After the first pass, subsequent
requests hit llama-server's prompt cache. The tight stdev (0.28 tok/s)
partly reflects cache hits producing uniform timings rather than genuine
stability under novel prompts. Real clinical use has higher prompt
diversity than a 10-prompt cycle.

**Dense Q4 comparison is from signoff notes, not JSONL.** The gemma3:4b
timeout and qwen2.5:7b failure cited in the comparison section come from
session signoff notes, not from a reproducible bench harness with
published raw data. A replicator cannot verify these numbers from the
provided artifacts.

**Workday simulation started at elevated swap.** Condition 3 ran after
the 1-hour sustained bench without rebooting. Starting swap was 6.2GB.
The +96MB delta reflects incremental pressure on an already-pressured
system, not the trajectory a practitioner would see from a fresh boot
through a full clinical day.

**Model quality is out of scope.** This finding measures throughput and
deployment envelope, not output quality. "Clinically coherent" is a
qualitative observation from reading the outputs, not a scored
evaluation. A separate quality bench (scored against reference outputs
by a BCBA) is needed before the clinical deployment claim is complete.

## Application

A sole proprietor (BCBA, teacher, small business owner) with an 8GB
M-series Mac buys a machine that can serve an 8B language model locally
with no subscription, no internet, no GPU. The model generates clinical
content (social stories, visual schedules, session notes) in 3-4 seconds
for 150-token responses while the application serves simultaneously.

The deployment cost: hardware they already own + a 1.1GB download +
`curl -sL subtract.ing/install.sh | bash`. Monthly operating cost after
that: electricity.

The stranded-buyer category (people who purchased M-series Macs for
cloud AI products that raised prices or degraded) have hardware that
runs Bonsai 8B at deployment-viable throughput. They do not know this.
The lookdown.tsv entry is how they find out.

# 1-Bit Wins Across Four Tiers Because the Bottleneck Differs by Tier

## Claim

Bonsai 8B Q1 (1.1GB) is deployment-viable on every hardware tier tested,
but the reason it wins differs by tier. 1-bit quantization eliminates
memory capacity as the binding constraint on all four platforms, exposing
the next bottleneck down — which is different on each:

  M2 Mac Mini 8GB:     memory capacity was the gate. Removed. Next: bandwidth (sufficient for 1 user).
  1660 Ti laptop 6GB:  VRAM capacity was the gate. Removed. Next: thermal throttle on laptop chassis.
  i7-1185G7 CPU 11GB:  capacity was never the gate. Win is fewer ops per token. Next: CPU compute.
  M1 Max Studio 64GB:  capacity was never the gate. Win is fewer bytes per token. Next: memory bandwidth.

Single-instance throughput on M1 Max varied 87-93 tok/s across bench
runs (87.1 in the scaling curve, 93.5 in the four-model sequential
comparison). The spread reflects warmup state and prompt cache
differences between runs, not measurement error.

The bottleneck taxonomy explains why the speedup ratio varies across
tiers rather than being a fixed property of the quantization.

## Measurement

Same model (Bonsai-8B-Q1_0.gguf, 1.1GB), same prompt set (10 clinical
ABA prompts, 150 max_tokens), same harness across all four platforms.

  Hardware              Chip          RAM/VRAM   tok/s (single)   Model size   Bottleneck
  M2 Mac Mini           Apple M2      8 GB       40.0             1.1 GB       bandwidth (swap irrelevant)
  Acer laptop           GTX 1660 Ti   6 GB       24.4             1.1 GB       thermal (11% throttle, recovers)
  Surface Pro 8         i7-1185G7     11 GB      8.0-9.3          1.1 GB       CPU compute
  Mac Studio            M1 Max        64 GB      87-93 tok/s      1.1 GB       memory bandwidth

### Efficiency comparison: Bonsai Q1 vs Qwen 2.5 7B Q4 (same CPU)

On the Surface Pro 8 (i7-1185G7, AVX2, no GPU):

  Model              File size   Generation tok/s   Prompt tok/s
  Bonsai 8B Q1       1.1 GB      9.3                8.7
  Qwen 2.5 7B Q4     4.4 GB      5.9                12.5

1.6x faster generation in 4.0x less memory. The generation advantage
comes from fewer bytes read per token. The prompt disadvantage (8.7 vs
12.5) reflects Q4's better arithmetic density during the compute-bound
prefill phase. Different bottlenecks at different inference stages.

Caveat: different model families, different training data, different
architecture. The comparison is practical (what fits and runs on this
hardware), not controlled. The memory ratio is clean. The throughput
ratio is confounded by architecture differences.

### Per-tier bottleneck evidence

**M2 Mac Mini (swap isolation).** 416-run sustained bench over 1 hour.
Swap grew from 737MB to 7,326MB. Throughput unchanged at 39.8 tok/s
mean. Behavior consistent with Metal-allocated model weights being
unaffected by system swap pressure. The OS pages out everything else;
the model stays resident. The constraint on M2 was memory capacity
(dense Q4 8B = 4.5GB, doesn't fit alongside application server in 8GB).
Bonsai at 1.1GB removes that constraint entirely.

**1660 Ti laptop (thermal).** 50-run bench: 24.4 tok/s. Sustained load:
24.6 → 22.0 tok/s over 15 minutes, stabilized near baseline at 24.3
after thermal equilibrium. The 11% dip is the laptop chassis thermal
envelope, not the GPU's compute limit. The constraint is heat dissipation
in a closed-lid form factor, not VRAM or compute.

**i7 CPU (compute).** Bonsai wins on generation (9.3 vs 5.9 tok/s)
because 1-bit reads fewer bytes per token from DDR4. But the absolute
speed (9.3 tok/s) is the lowest across all tiers because x86 CPU
matrix multiply is the binding constraint, not memory bandwidth. 1-bit
helps (fewer ops) but doesn't change the regime.

**M1 Max Studio (bandwidth).** 93.5 tok/s single instance. Under
concurrent load, aggregate throughput plateaus at ~124 tok/s regardless
of instance count (see finding 10). The constraint is unified memory
bandwidth. 1-bit maximizes tokens per byte read. Bonsai generates
at comparable or faster speeds than phi4-mini Q4 (88.4 tok/s) and
Qwen 2.5 7B Q4 (~88 tok/s) despite having more parameters. The two
Bonsai measurements on M1 Max (87.1 in the scaling curve, 93.5 in
the four-model comparison) bracket the dense Q4 models — the
variation is from warmup and cache state, not measurement error.

## Mechanism

1-bit quantization represents each weight as {-1, 0, +1}. The
arithmetic shifts from multiply-accumulate (FP16/Q4) to addition and
subtraction (Q1). Two consequences:

1. Model file size drops from ~4.5GB (Q4 8B) to 1.1GB (Q1 8B).
   This removes memory capacity as a constraint on hardware where
   it was previously binding (M2 8GB, 1660 Ti 6GB VRAM).

2. Bytes read per token drop proportionally. On bandwidth-bound
   platforms (M1 Max, M2), this translates directly to higher tok/s.
   On compute-bound platforms (i7 CPU), the benefit is smaller because
   the bottleneck is arithmetic throughput, not memory reads.

The bottleneck taxonomy is a consequence of removing the top constraint.
When capacity stops being the limit, whatever was second in line becomes
first. That second-in-line constraint is hardware-specific.

## Confounds

**Different model families in the efficiency comparison.** Bonsai 8B
and Qwen 2.5 7B have different architectures, training data, and
parameter counts. The throughput comparison is practical, not controlled.
A same-model different-quant comparison (Bonsai at Q1 vs Q4) would
isolate quantization from architecture, but Q4 Bonsai GGUF does not
exist.

**Single prompt set.** All measurements use the same 10 clinical ABA
prompts. Different prompt shapes (long context, code generation,
multilingual) may shift the bottleneck balance.

**Thermal measurement on 1660 Ti is chassis-specific.** A desktop
1660 Ti with active cooling would produce different thermal behavior.
The throttle finding is for this laptop chassis, not for the GPU
architecture.

**M2 swap isolation mechanism is attributed, not measured.** The
observation (throughput stable under 10x swap growth) is empirical.
The mechanism (Metal memory pinning) is inferred from Apple Silicon
architecture documentation, not from direct measurement (vmmap,
wired page counts).

## Prior work

bitnet-log (03-git, frozen 2026-04-25) tracked the convergence of
two capability gates for local inference: weight compression (BitNet
ternary weights) and KV cache compression. The weight gate thesis:
ternary weights at 1.58 bits per parameter enable large models on
consumer hardware by shifting the binding constraint from memory
capacity to memory bandwidth. This finding validates that thesis
empirically. The cache gate (KV compression for long context) remains
open and is tracked separately.

# Aggregate Inference Throughput Plateaus at a Hardware-Determined Ceiling

## Claim

On M1 Max Mac Studio (64GB, ~400 GB/s memory bandwidth), aggregate
inference throughput for Bonsai 8B Q1 plateaus at ~124 tok/s regardless
of how many concurrent instances serve. Adding users costs per-user
latency but gains no aggregate throughput beyond 4 concurrent. The
ceiling is the memory bandwidth, not compute, not RAM capacity, not
the model.

This means capacity planning for 1-bit inference reduces to a single
question: what is the memory bandwidth of the target hardware?

## Measurement

10 Bonsai 8B Q1 instances (ports 9100-9109) on M1 Max Mac Studio,
all loaded from the same 1.1GB GGUF via mmap (shared physical pages
for model weights, separate KV caches per instance). Prompt: "write
3 steps for hand washing", 100 max_tokens. Concurrency levels tested:
1, 2, 4, 6, 8, 10.

  Concurrent   Per-user tok/s   Aggregate tok/s   Wall time (100 tok)
  1            87.1             87.1              1.2s
  2            47.1             94.2              1.9s
  4            31.2             124.8             2.7s
  6            20.3             121.9             4.1s
  8            15.6             124.5             5.2s
  10           12.3             122.8             6.9s

Aggregate plateaus at 124 tok/s from 4 concurrent onward. Per-user
throughput divides linearly: at 10 users, each gets approximately
1/10 of the bandwidth, producing ~12 tok/s (an 8-second response
for 100 tokens).

The knee is at 4 concurrent. Below 4, aggregate grows with each
user (bandwidth not yet saturated). At 4 and above, aggregate is
flat — the bus is full.

Data points at 12 and 16 concurrent showed anomalously high aggregate
(199, 230 tok/s), attributed to llama-server prompt caching on the
identical repeated prompt. These are excluded pending rerun with
varied prompts.

### Shared mmap enables the scaling

10 Bonsai instances do not consume 10x the model memory. The OS maps
the same 1.1GB file into each process's address space, sharing the
physical pages. Per-instance marginal cost is the KV cache (~200-500MB
depending on context length). This is why 10 instances fit in 64GB
without swap pressure — total memory is approximately
1.1GB (shared weights) + 10 x 300MB (KV caches) = ~4.1GB, not 11GB.

Without mmap sharing, the scaling curve would hit a memory capacity
wall before hitting the bandwidth ceiling. Mmap is the mechanism
that exposes bandwidth as the true constraint.

### Four-model concurrent comparison

For context, the bandwidth ceiling was also observed under a different
condition: 4 different models (Gemma 26B Q4, Bonsai 8B Q1, Phi4-mini
Q4, Qwen 2.5 7B Q4) serving simultaneously through h2c (squared
proxy):

  Model                      Sequential tok/s   Concurrent tok/s
  Gemma 26B Q4 (15.9GB)      55.8               21.6
  Bonsai 8B Q1 (1.1GB)       93.5               29.6
  Phi4-mini Q4 (2.3GB)       88.4               25.5
  Qwen 2.5 7B Q4 (4.4GB)    ~88                20.6

Aggregate concurrent: ~97 tok/s (lower than the N-Bonsai ceiling
because four different weight sets compete for bandwidth, while
N-Bonsai shares one weight set via mmap).

## Mechanism

Inference on 1-bit models is memory-bandwidth-bound, not compute-bound.
Each token requires reading model weights from memory. With 1-bit
weights, the arithmetic per byte is minimal (addition/subtraction
instead of multiply-accumulate), so the GPU finishes the math before
the next chunk of weights arrives. The memory bus is always the
bottleneck.

Under concurrency, multiple instances read weights simultaneously.
When the instances share the same model via mmap, they share the
same physical memory pages and compete only for bus bandwidth, not
for memory capacity. The aggregate throughput saturates at the bus
bandwidth divided by bytes-per-token.

On M1 Max with ~400 GB/s unified memory bandwidth, the observed
ceiling of ~124 tok/s implies approximately 3.2 GB read per token
under concurrent load. However, this ratio is not a hardware-independent
constant. M2 at ~100 GB/s produces 40 tok/s single-user (implying
2.5 GB/tok), not the 31 tok/s the M1 Max ratio would predict. The
difference reflects cache hierarchy, concurrency effects (M1 Max
measured under contention, M2 under single-user), and memory subsystem
specifics. The bandwidth ceiling is real; the bytes-per-token ratio
varies by platform and load.

## Confounds

**N=1 per concurrency level.** Each concurrency level was measured
once. The "plateau at ~124" rests on four data points (124.8, 121.9,
124.5, 122.8) with no repeats, no error bars, no confidence intervals.
The plateau is visually convincing but statistically uncharacterized.
A rerun with 5 passes per concurrency level would strengthen or
invalidate the claim.

**Prompt caching.** The bench used the same prompt for all instances
at all concurrency levels. llama-server's prompt cache may have
inflated throughput on repeated runs by skipping prefill. The kept
data points (1-10 concurrent) are also subject to this effect, but
per-user generation throughput is post-prefill and the aggregate
is wall-clock dominated by generation, bounding the cache effect.
The 12/16 exclusion reflects visible anomaly, not a claim that
1-10 is cache-free. Cache-free verification with varied prompts
is the next bench.

**Concurrency above 10 reused ports.** Concurrency levels 12 and 16
reused ports modulo 10, meaning some llama-server instances served
two simultaneous requests. With -np 1 these requests serialize within
the instance, producing unreliable concurrent measurements. Excluded
pending a rerun with sufficient instances per concurrency level.

**Single hardware platform.** The bandwidth ceiling is specific to
M1 Max at ~400 GB/s. The ratio aggregate_tok/s / memory_bandwidth
should be testable on other platforms (M2 at ~100 GB/s, 1660 Ti at
~288 GB/s) to confirm it's a general property. Not yet measured.

**No varied prompt set.** A single prompt means no variance from
different prompt lengths or generation patterns. A varied prompt
set would produce a distribution of per-user tok/s rather than
near-identical values.

## Application

The scaling curve is a capacity planning instrument. Measured
ceilings by platform:

  M1 Max 64GB (400 GB/s):  ~124 tok/s aggregate, 10 users at 12 tok/s each
  M2 8GB (100 GB/s):        40 tok/s single user (concurrent not tested)
  1660 Ti 6GB (288 GB/s):   24 tok/s single user (thermal ceiling)
  i7 CPU (51 GB/s DDR4):    8-9 tok/s single user (compute ceiling)

The deployment question: how much bandwidth does the hardware have,
and how many concurrent users at what per-user floor? The bytes-per-
token ratio varies by platform and load (2.5-3.2 GB/tok observed),
so the ceiling must be measured per platform, not extrapolated from
a single ratio. The bandwidth determines the ceiling; the exact
ceiling requires a bench run.
